## 20. When to Tune?
- SQL 튜닝이 필요한 시점과 느린 쿼리의 원인 핵심 정리

## 1️⃣ SQL 튜닝은 일회성이 아니다
- SQL 튜닝은 **지속적인 과정**
- 쿼리 작성 시점부터 튜닝 고려 필요
- 작성 이후에도 환경 변화에 따라 재튜닝 필요


## 2️⃣ 언제 튜닝해야 하는가?

### ✔ 1. 쿼리 작성 단계
- 개별 쿼리는 1초 미만이라 느려 보이지 않을 수 있음
- 하지만 시스템 전체에서는 수천 개 쿼리가 동시에 실행됨
- 작은 비효율이 누적되면 전체 DB 성능 저하 발생
- 👉 작성 시점부터 튜닝 습관화 필요

### ✔ 2. Top Consuming Queries 발생 시
- 전체 쿼리의 **5% 이하가**
- 전체 자원의 **80%를 사용하는 경우가 많음**
- 통계를 통해 쉽게 조회 가능
- 👉 상위 소비 쿼리 위주로 집중 튜닝

### ✔ 3. Bad SQL 발견 시
- 비효율적인 SQL 구조
- 실행계획이 비정상적인 쿼리
- 👉 구조 개선을 통해 성능 향상 가능

### ✔ 4. 사용자 불만 발생 시
- 화면 멈춤
- 응답 지연
- “어제는 괜찮았는데 오늘은 느림”
- 👉 환경 변화 가능성 조사 후 튜닝


## 3️⃣ SQL이 느려지는 주요 원인

### 1. 구조적 변경
- 테이블 구조 변경
- 인덱스 생성/삭제
- 👉 실행계획 변화 발생

### 2. 데이터 양 변화
- 테이블이 커지거나 작아짐
- 통계 기반 실행계획 변경
- 👉 CPU 사용량, 대기 시간 증가 가능

### 3. 애플리케이션 변경
- Bind 변수 미사용
- 데이터 타입 변경 (숫자 → 문자 등)
- 👉 하드파싱 증가 및 실행계획 변화

### 4. 통계 노후화
- 최신 통계 미수집
- 잘못된 실행계획 생성

### 5. DB 업그레이드
- 대부분 개선되지만 일부 쿼리는 악화 가능

### 6. DB 파라미터 변경
- PGA, SGA 크기 변경 등
- 메모리 부족 시 성능 저하

### 7. OS / 하드웨어 변경
- 신규 장비 도입
- OS 변경
- 일부 쿼리 성능 저하 가능



## 🎯 핵심 메시지

- 소수의 쿼리가 대부분의 자원을 사용한다.
- 튜닝은 사후 대응이 아니라 **예방적 활동**이다.
- 작성 단계 + 정기 점검 + 사용자 피드백 기반 튜닝이 필요하다.
- 환경 변화는 언제든 실행계획을 바꿀 수 있다.

## 21. What is Bad SQL?

## 1️⃣ Bad SQL이란?

- 같은 결과를 반환하더라도  **불필요하게 많은 자원을 사용하는 SQL**
- 또는 **성능이 좋지 않은 SQL (Poorly Performing SQL)**

👉 필요 이상으로 10배~100배 더 많은 자원을 사용할 수 있음


## 2️⃣ 성능을 결정하는 4가지 핵심 요소

### 1. Parsing Time (파싱 시간)
- 쿼리 문법 검사
- Private SQL Area 할당
- 실행계획 생성
- 과도한 파싱 = 성능 저하
- 👉 불필요한 하드파싱은 Bad SQL 원인

### 2. I/O (디스크 읽기)
- 데이터를 얻기 위해 읽는 블록 수
- 1행을 얻기 위해 1,000블록을 읽을 수도 있음
- 인덱스/쿼리 구조에 따라 차이 발생
- 👉 과도한 블록 읽기 = Bad SQL

### 3. CPU Time
- Join, Sort, 연산 등은 CPU 사용
- 불필요한 조인/정렬 → CPU 낭비
- 👉 과도한 CPU 사용 = Bad SQL

### 4. Wait Time (대기 시간)
- CPU 차례 대기
- 네트워크 대기
- Lock 대기 등
- DB TIME = 실행 시간 + 대기 시간
- 👉 대기 시간이 길면 튜닝 필요

## 3️⃣ Bad SQL의 공통 특징

- 과도한 Parsing
- 과도한 I/O
- 과도한 CPU 사용
- 과도한 Wait 발생

대부분의 정상 쿼리는 **1초 이내 반환**
몇 초~몇 분 걸리면 반드시 점검 필요

## 4️⃣ Bad SQL이 발생하는 원인

- 잘못된 스키마 설계
- 비효율적인 코딩
- 나쁜 실행계획(Execution Plan)

## 22. Effective Schema Design

## 🎯 왜 스키마 설계가 중요한가?

- 성능 튜닝의 첫 단계는 **스키마 설계**
- 스키마를 잘 설계하면 수백 개의 쿼리를 별도 튜닝하지 않아도 됨
- 잘못 설계하면 이후에 많은 성능 문제 발생
- 처음 설계가 매우 중요 (나중 수정은 어렵고 비용 큼)


## 1️⃣ 데이터 타입을 신중하게 선택하라

### ✔ 필요한 만큼만 지정
- `varchar2`는 가변 길이 → 실제 저장된 데이터 크기만 사용
- `char`는 고정 길이 → 지정한 크기만큼 항상 사용

### ✔ 주의할 점
- 필요 이상으로 크게 지정하면 데이터 무결성 문제 발생
- 숫자/날짜를 `varchar2`로 저장하지 말 것 (성능 저하)
- 컬럼 목적에 맞는 데이터 타입 사용

## 2️⃣ PK와 FK의 데이터 타입은 반드시 동일하게

- 부모/자식 키는 **정확히 같은 데이터 타입 사용**
- 다르면 조인 시 암묵적 형변환 발생
- 형변환 → CPU 사용 증가 → 성능 저하

## 3️⃣ 제약조건을 정확히 정의하라

- Primary Key, Foreign Key 명확히 정의
- Check Constraint 사용하여 데이터 중복 방지
- 제약조건은 실행계획 생성 시 중요한 정보로 활용됨

## 4️⃣ 정규화를 올바르게 적용하라

- 데이터 중복 감소
- 테이블 크기 감소
- 유지보수 용이
- 작은 테이블 간 조인 → 성능 향상

## 5️⃣ 적절한 테이블 유형 선택

Oracle 테이블 유형:
- Heap-Organized Table
- Index Clustered Table
- Hash Clustered Table
- Index-Organized Table (IOT)
- External Table

👉 상황에 맞는 테이블 타입 선택 시 성능 향상

## 6️⃣ 클러스터링 활용

- 자주 조인되는 테이블을 물리적으로 가깝게 저장
- 반복 조인 성능 개선 가능

## 7️⃣ 인덱스를 적극 활용하되 신중하게

- 인덱스는 가장 강력한 성능 향상 수단
- 다양한 인덱스 타입 존재
- 단, 인덱스 생성에는 DML 비용 증가

## 8️⃣ Index-Organized Table(IOT) 활용

- 테이블을 인덱스 구조에 저장
- 작은 테이블 + 자주 조회되는 경우 효과적

> TB_RT_POS table 적용 고려? -> 알아보기

## 🔥 핵심 메시지

> 좋은 스키마 설계 = 가장 강력한 성능 튜닝

- 잘 설계하면 이후 튜닝 비용 최소화
- 잘못 설계하면 저장공간, 성능, 비즈니스 문제 발생
- 스키마 설계는 단순 구조 문제가 아니라 **성능 전략의 출발점**

## 23. Table Partitioning

## 1️⃣ 왜 Partitioning이 필요한가?

- 대용량 테이블에서 **Full Table Scan 비용이 매우 큼**
- **범위 검색(range search) 시 인덱스가 비효율적일 수 있음**
- 결과적으로 전체 블록을 읽게 되어 성능 저하 발생

## 2️⃣ Partitioning이란?

- 하나의 테이블을 **물리적으로 여러 개의 작은 파티션으로 분할**
- 특정 컬럼 기준(예: 날짜)으로 나눔
- 데이터는 조건에 맞는 파티션에 자동 저장

## 3️⃣ 성능 향상 원리

- 쿼리가 특정 파티션에만 해당하면
  → **전체 테이블이 아니라 해당 파티션만 읽음**
- 예: 100개 파티션 중 1개만 조회 → 최대 100배 성능 개선 가능
- 여러 범위 조회 시에도 필요한 파티션만 읽음

## 4️⃣ 언제 사용하는가?

- 테이블이 매우 큰 경우
- 특정 컬럼이 자주 WHERE 조건에 사용되는 경우
- 날짜 기반 범위 검색이 많은 경우

## 5️⃣ 인덱스 Partitioning

- 인덱스도 파티셔닝 가능
- 대용량 인덱스 검색 범위 축소
- DML 부담 완화 가능

## 6️⃣ 성능 외 장점

- 특정 파티션 단위 백업/복구 가능
- 병렬 처리 최적화 가능

## 🎯 핵심 메시지

> 대용량 테이블 + 자주 사용하는 필터 컬럼이 있다면  
> 해당 컬럼 기준으로 Partitioning을 고려하라.

Partitioning은 특히 **범위 검색 성능 개선에 매우 강력한 기법**이다.

## 7️⃣ 개인 궁금증
- 범위기반 조회는 인덱스를 B+Tree 구조로 생성해서 극복하지 않나?

### **B+Tree 인덱스 vs Table Partitioning**

#### B+Tree 인덱스
- 정렬된 리프 노드 구조
- 범위 검색 시 시작 지점 빠르게 탐색 (O(log N))
- 리프 노드 순차 스캔 가능
- 👉 **논리적 탐색 속도는 매우 빠름**

**하지만 실제 I/O는 다르다**
- 인덱스 리프에는 `Key + RowID` 저장
- 실제 데이터는 **테이블 블록**에 존재
- RowID를 통해 해당 블록으로 이동해야 함 => **결국 물리적 I/O 비용이 커진다.**

**범위가 커지면?**
- 많은 RowID 발생
- 블록 랜덤 접근 증가 => **Random I/O 폭증**
- 오히려 Full Table Scan이 더 빠를 수 있음

**Partitioning의 역할**

- 테이블을 **물리적으로 여러 개로 분리**
- 특정 범위 조회 시 해당 파티션만 읽음
- 읽어야 할 물리 블록 자체를 줄임
- 큰 범위를 작은 물리 범위로 분할

**🎯 핵심 차이**
| 구분 | 역할 |
|------|------|
| B+Tree 인덱스 | 논리적 탐색을 빠르게 함 |
| Partitioning | 물리적 읽기 범위를 줄임 |
**🔥 한 줄 정리**
> 파티셔닝은 **읽어야 할 데이터 양 자체를 줄인다**
